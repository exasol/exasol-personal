## 2. Embedded registry in `internal/runtimeartifacts`

- [x] 2.1 Add a new file (e.g. `embed.go`) in `internal/runtimeartifacts` with a package-level `map[string][]byte` registry, an exported `Register(resourceID string, data []byte)` that no-ops on empty/nil `data`, and an internal lookup function.
- [x] 2.2 Add `Embed bool` to `cacheIndexEntry`, copied from `def.Embed` in `resolveEntry` — the same treatment `Extract` already gets — so `refresh()` doesn't need `def` threaded in separately.
- [x] 2.3 Modify `Manager.refresh()` (`manager.go`) to branch strictly on `entry.Embed`, with no fallback: if `true`, look up the registry and treat a miss as a hard error (`no embedded data registered for resource %q`) rather than calling `m.fetch()`; if `false`, call `m.fetch()` exactly as today. On a hit, write the looked-up bytes to `entry.ArtifactPath` (same slot `fetch()` would populate). The existing `if entry.Extract { m.extract(*entry) }` step runs unmodified after either branch.
- [x] 2.4 Do not add any checksum re-verification for the embedded-hit path — `sha256` is a build-time-only concern for embedded resources (see §3).
- [x] 2.5 Unit tests: `Manager` resolves an `embed: true` resource exclusively from the registry when `Register`ed; returns a hard error (not a network fetch) when `embed: true` and nothing is registered; a resource without `embed: true` never consults the registry even if something happens to be registered under the same ID; extraction runs identically for an embedded-sourced artifact and a network-fetched one of the same archive format. Since `Register`/lookup are plain functions over a package-level map, tests can call `Register` directly to simulate embedded content — use `t.Cleanup` (or distinct resource IDs) to avoid cross-test interference given the shared mutable map.
- [x] 2.6 Simplified `FileSource.Fetch` to redirect to any existing local path uniformly (directory, archive, or bare file), removing an over-restrictive check that rejected anything but a directory or a recognized archive extension. Archive-vs-bare-file correctness is enforced downstream by `Manager.extract()`'s `Extractor` selection (which only runs when `extract: true`), not by `FileSource` itself — this is what makes the `extract: false` + bare-binary `file://` override (documented in `doc/development.md`) work.

## 5. Taskfile / release pipeline

- [x] 5.1 Replaced `local-runner-placeholders`/`local-runner-stage` with `resource-embed-placeholders` (`-placeholder-only`, no network) and `resource-embed-stage` (real fetch, `TARGET_GOOS`/`TARGET_GOARCH`-parameterized). **Refinement made during implementation**: the tool needed an explicit `-placeholder-only` mode (not in the original design) — without it, running it as a `tests-unit`/`lint-golang` prerequisite would trigger a real network fetch whenever the host happens to match a declared platform (e.g. on a Mac), which the old two-task split had specifically avoided. `-placeholder-only` also never overwrites a file that already exists, so a developer who's staged the real runner doesn't have it silently blanked out by a later test/lint run.

- [x] 11.3 Fixed a real cross-compilation bug: a `//go:generate` directive's subprocess inherits ambient `GOOS`/`GOARCH`, so `GOOS=darwin GOARCH=arm64 task generate`/`task build` cross-compiled the generator tool itself instead of just its target, causing "exec format error." Fixed by having `generate`, `resource-embed-stage`, and `resource-embed-placeholders` `unset GOOS GOARCH` before invoking `go generate`/the generator directly; `task build` now stages the requested target's resource via `task resource-embed-stage TARGET_GOOS=... TARGET_GOARCH=...` before its own (still cross-compiling) `go build` step. Verified against real `darwin/arm64` and `windows/amd64` targets, and the unmodified host build.
- [x] 11.4 `doc/development.md` updated: the bare-file `file://` override section rewritten for the corrected `FileSource` behavior, and the "Building Without Task" cross-compilation example fixed to use the generator's own `-goos`/`-goarch` flags rather than `GOOS`/`GOARCH` env vars (which would cross-compile the `go run` invocation itself).
