## Context

The resource manager (`internal/runtimeartifacts.Manager`) already fetches, caches, verifies, and extracts runtime dependencies like `tofu` from `resources.yaml`. The VM runner (`exasol-local-runner`) is the one exception: it's fetched at build time by a bespoke tool (`tools/localrunner`), baked into the binary via hand-written `//go:embed` files (`assets/localruntimebin/`), and — as of a recently-merged colleague change — reconciled into the deployment directory by a parallel, domain-specific semantic-versioning system (`internal/localruntime.reconcileRunner`) that keeps a physical, atomically-replaced copy of the runner inside each deployment's directory, upgrading or refusing to upgrade it based on `blang/semver` comparison against the currently-embedded runner.

This change unifies the "how do I get these bytes" half of that (embedding becomes a resource source, same as a download) while deliberately preserving the "is this version change safe for this specific deployment's VM" half, which is genuinely domain-specific and doesn't belong in a generic resource manager.

## Goals / Non-Goals

**Goals:**
- Make "embedded in the binary" a declarative, first-class resource source (`embed: true` in `resources.yaml`), usable by any resource, not just the VM runner.
- Make build-time embedding a generic, `go generate`-driven feature with zero resource-specific `.go` files checked into git.
- Preserve the existing semver-based runner-compatibility behavior (upgrade on compatible bump, refuse downgrade, gate major version changes, migrate legacy unversioned runners) as domain logic in `internal/localruntime`, re-based on a persisted version marker instead of a physical installed copy.
- Support a documented `file://` override in `resources.yaml` for fast local iteration on the embedded resource, reusing the existing `FileSource` — no new mechanism needed for this.

**Non-Goals:**
- Per-platform `embed` granularity (resource-level `embed: true` only).
- Repackaging or recompressing an embedded artifact before embedding — only justified if some future embeddable resource's real archive isn't already minimal (not true today: `exasol-local-runner`'s release zip contains only the `launcher` file, confirmed by reading the `exasol-local-vm` build scripts).
- zstd support — the design accommodates it later via the existing `Extractor` list, but nothing about this change requires it.
- A compiler-enforced guarantee that the generator tool can never see embedded data (accepted as a convention, not tooled).
- Any live, no-rebuild-required runtime override mechanism for embedded resources — see Decision on `file://` overrides below.

## Decisions

### `resources.yaml` schema: resource-level `embed: true`
A resource opts into embedding by setting `embed: true` on its `ResourceDefinition` in `spec.go`, alongside the existing `extract` field. It reuses the same `artifact` map's `url`/`sha256` that network fetching already uses — one artifact entry serves both build-time embedding and, for anything not embedded, ordinary runtime fetching. `embed: true` applies uniformly to every platform declared under that resource's `artifact:` map.

*Alternative considered:* a separate `embedded://` URL scheme, so an artifact's source could be distinguished by its URL rather than a flag. Rejected in favor of the single flag: it keeps one canonical definition of "how to get this resource" per platform, rather than two parallel ones that would need to stay in sync.

### Runtime dispatch lives inside `internal/runtimeartifacts`, unrenamed
`internal/runtimeartifacts` keeps its existing name, location, and contents, and gains a small `Register`/lookup registry (a package-level `map[string][]byte`; `Register` no-ops on empty data). Generated per-platform files live under `assets/resources/generated/`, alongside the existing `assets/resources/resources.yaml`/`resources.go`, since this is asset data, not `runtimeartifacts`-internal machinery. `assets/resources/generated` imports `internal/runtimeartifacts` to call `Register`; `internal/runtimeartifacts` never imports it back, so there's no cycle. `cmd/exasol/main.go` blank-imports `assets/resources/generated` to trigger the `init()`s — one line, mirroring the stdlib `time/tzdata` pattern.

*Alternative considered:* renaming today's `runtimeartifacts` to `internal/resources` and repurposing the `runtimeartifacts` name for a thin "activation shim" package, to keep the generated-data import out of the main engine package. Rejected as a larger, repo-wide rename for a benefit (a slightly cleaner import graph) already achievable by simply placing the generated data under `assets/resources/generated` instead of nesting it inside `runtimeartifacts`.

### Resolution dispatch is a strict `embed` either/or
`resolveEntry` copies `def.Embed` onto `cacheIndexEntry.Embed`, the same way it already copies `def.Extract` onto `cacheIndexEntry.Extract`, so `refresh()` doesn't need `def` threaded in separately. `refresh()` then branches purely on `entry.Embed`: if `true`, it looks up the registry and treats a miss as a hard error (`no embedded data registered for resource %q`); if `false`, it calls `m.fetch()` against the network `sources` list exactly as today. Both branches converge back into the same unmodified `if entry.Extract { m.extract(*entry) }` step — no new decompression/compression code either way, reusing the same `Extractor`s already used for network-fetched archives (so a future zstd resource is served automatically once a zstd `Extractor` exists for its own network-fetch sake). The build-time generator gets network-sourced resolution for an `embed: true` resource by constructing a local copy of the resource's `ResourceDefinition` with **both** `Extract` and `Embed` forced to `false` before calling `Get`.

*Alternative considered:* checking the embedded registry for `embed: true` resources and silently falling back to the network `sources` list on a miss. Rejected because it makes the generator's "always fetch for real" property depend on an *absence* (the registry happening to be empty in that process) rather than a deliberate statement, and because a missing registration for an `embed: true` resource in the shipped binary would silently degrade to a network fetch instead of surfacing a real build problem loudly. Forcing `Embed: false` locally in the generator is also a strictly stronger guarantee than relying on the registry being empty: even if the generator process ever accidentally linked the generated registry package, `refresh()` still wouldn't consult it for that call.

### Generated files are build-tag-gated, one per resource × declared platform, always present
For every `embed: true` resource, each declared platform gets its own build-tag-gated generated file with real data, and every other platform the project builds for gets its own generated file with no data — every platform is covered by a distinct file, and generating one never touches another. This satisfies Go's "package needs ≥1 file matching build constraints" requirement uniformly, without a hand-maintained catch-all stub, and lets goreleaser's existing per-target `hooks.pre` (`.goreleaser.yaml` already runs `task resource-embed-stage TARGET_GOOS="{{.Os}}" TARGET_GOARCH="{{.Arch}}"` per release target, likely in parallel) regenerate each target's file independently and safely. All of `assets/resources/generated/` is gitignored; nothing resource-specific is ever checked in.

*Alternative considered:* regenerating a single shared file fresh per invocation, with no build tags, so its content always matches whatever target was last requested. Rejected because it would race against itself across goreleaser's concurrent per-target builds sharing one file path, and it removes the compiler's guarantee that a forgotten regeneration can't silently embed the wrong platform's bytes into a local cross-compile.

A related correctness issue for cross-compiling (`GOOS=... GOARCH=... task build`, or `task generate`): a `//go:generate` directive's `go run`/`go tool` subprocess is itself subject to whatever `GOOS`/`GOARCH` are set in the invoking shell, so an ambient cross-compilation target causes the generator tool to be cross-compiled too, and it then fails to execute at all ("exec format error") instead of generating anything. `generate`, `resource-embed-stage`, and `resource-embed-placeholders` in `Taskfile.yml` each `unset GOOS GOARCH` before invoking `go generate`/the generator directly, so the directive always runs the tool natively regardless of what target the surrounding `task build` is cross-compiling for; `task build` stages the requested target's resource first (`task resource-embed-stage TARGET_GOOS=... TARGET_GOARCH=...`) before its own `go build` step, which is what still receives the real `GOOS`/`GOARCH`.

### Generator tool forces `Extract`/`Embed` false on a local copy, no new `Manager` method
The generator needs the raw, pre-extraction artifact bytes, fetched over the network regardless of what `resources.yaml` says about embedding. It constructs a local copy of the resource's `ResourceDefinition` with `Extract` and `Embed` both forced to `false` before calling the existing `Get` — `fetch()`'s checksum verification is unconditional on `Extract`, and `resolveEntry` already returns the raw artifact path when `Extract` is false, so this needs no new Manager API, just a clearly-commented call site explaining both overrides. The new tool (`tools/resourceembedder`) replaces `tools/localrunner` outright, not alongside it.

*Alternative considered:* adding a `GetRaw` method to `Manager` that does this internally. Rejected to keep `Manager`'s public API surface exactly as it is today, for a need that has exactly one caller.

### VM runner resolves directly from the shared cache on every invocation
`internal/localruntime.runnerCommand`/`runnerCommandWithOutput` already set `cmd.Dir` to the deployment's working directory while invoking the runner by absolute path, so the runner is resolved via the resource manager and executed directly from wherever it resolves in the shared cache, exactly like `tofu` — the binary never needs to live inside the deployment directory.

The semver-based compatibility behavior (upgrade on compatible bump, no downgrade, major-version gate, legacy-runner migration, the `EXASOL_LOCAL_FORCE_RUNNER_RECONCILIATION` escape hatch) is preserved as domain logic in `internal/localruntime`, because it protects something real and independent of *how* the runner's bytes are obtained: a long-lived VM's on-disk data/protocol may not tolerate an unplanned version jump. It's re-based on a small persisted version marker (a version string, not a binary) recorded in the deployment's own state, compared against the currently-resolved runner's reported version (via running `<resolved-path> version`, as `readRunnerVersion` already does) on each prepare/start.

`internal/deploy/backend.go`'s `newDeploymentBackend` constructs the `*runtimeartifacts.Manager` for the local backend the same way it already does for `tofu`, and `newLocalBackend` wraps it together with the deployment into a single exported `internal/localruntime.Runtime` struct (via `localruntime.New(deployment, manager)`), stored once on `localBackend` and used for every operation (`Prepare`, `RunCommand`, `ReadState`, `Status`, `HealthCheck`, `Stop`, `Destroy`) as methods on that value — VM-runner-specific mechanics (SSH keys, runner state parsing, command invocation) stay inside `internal/localruntime`, consistent with the `local-runtime-boundaries` spec's delegation requirement, while Manager construction stays at the backend-selection layer uniformly for every backend.

Because `exasol-local-runner` is `embed: true`-only with no artifact declared for non-macOS platforms, there is no way for a non-Mac host (including every Linux CI runner) to resolve a real runner at all — a gap the old bespoke embedding didn't have, since tests could stage a fake binary directly into the deployment directory instead. `Runtime.resolveRunnerPath` checks an `EXASOL_LOCAL_RUNNER_OVERRIDE_PATH` environment variable first and, if set, returns it directly, bypassing the `Manager` entirely; this is the mechanism integration tests and non-macOS contributors use to exercise the rest of the local-deployment flow against a fake or locally-built runner.

*Alternatives considered:* keeping the physical per-deployment copy and its atomic candidate-replace machinery, only swapping the source of the "candidate" bytes from the `localruntimebin` var to the resource manager. Rejected once we confirmed the manager's cache resolution makes a separate installed copy redundant — `cmd.Dir` already decouples where the binary lives from where it executes, so there's nothing left for a per-deployment copy to do. Also considered: constructing the `Manager` inside `internal/localruntime` itself, to keep Manager construction next to the component that uses it. Rejected because the `local-runtime-boundaries` delegation is about where VM-runner-specific mechanics live, not about who constructs a generic, cross-cutting dependency shared with other backends — constructing it uniformly at the backend-selection layer, as `tofu` already does, is more consistent with that boundary, not less.

### `file://` override for local development reuses the existing `FileSource`
`resources.yaml`'s `url` field for any resource — including `embed: true` ones — already supports `file://` (or a bare local path) via the existing `FileSource`, which redirects straight to the local path without copying it into the cache or verifying it against `sha256`. This is the documented way to iterate on the embedded VM runner without touching the network resource. `FileSource` itself is fully permissive about what the path points at (a directory, a supported archive, or a bare file) — whether that's usable is entirely a function of the resource's own `extract`/`resource_path` settings, not a restriction `FileSource` enforces:
- For `extract: true` resources (the committed `exasol-local-runner` entry), extraction still runs against whatever `url` resolves to, so it needs to be something a registered `Extractor` can unpack (a `.zip`/`.tar.gz`/`.tgz`), with `resource_path` picking the file out of the extracted result.
- Setting `extract: false` alongside the override (and dropping `resource_path`) instead lets `url` point directly at a bare binary — e.g. a locally-built `launcher` — with no zipping step needed; the bare file is embedded and resolved exactly as given. This is the primary override use case, since it lets a developer iterate on the launcher without repackaging it each time.
- `sha256` must still be present in the YAML entry for parse-time validation to pass (any non-git source requires one), but it is not verified against the local file's content — the existing value can be left in place; it's inert for a `file://`-sourced artifact.
- For an `embed: true` resource, this override takes effect through the *generator*'s next run — the registry always wins over the network/file sources at runtime once something is registered (per the strict either/or above), so a `file://` edit requires re-running the generator and rebuilding `exasol` to take effect; it is not a live, no-rebuild override.

*Alternative considered:* a dedicated override mechanism (e.g. a CLI flag or environment variable pointing at a local runner build), as `tools/localrunner`'s old `--runner-path` flag did. Rejected as unnecessary — `FileSource` already provides this for any resource without new code, so a purpose-built override would just be a second way to do the same thing.

*Note:* this local-iteration override is distinct from `EXASOL_LOCAL_RUNNER_OVERRIDE_PATH` (see the "VM runner resolves directly from the shared cache" decision above) — that env var bypasses the resource manager entirely at *runtime*, for tests and non-macOS contributors who have no embedded artifact to resolve at all; the `file://` override instead changes what the *generator* embeds on its next run, still going through the manager and the registry normally.

## Risks / Trade-offs

- **[Risk]** The generator tool could someday accidentally import `assets/resources/generated`. → **Mitigation**: largely addressed by construction, not just convention — the generator forces `Embed: false` on its local `ResourceDefinition` copy before every `Get` call, so `refresh()` never consults the registry for that call regardless of what the generator process happens to have linked. No CI/lint guardrail beyond that (deliberately, per discussion).
- **[Risk]** Removing the physical per-deployment runner copy changes the shape of the major-version-mismatch behavior: previously, a mismatch meant "keep using the already-installed older runner" (a real fallback binary existed); now there is no older runner to fall back to, since the manager only ever resolves the one currently available. See Open Questions.
- **[Trade-off]** Embedding the generator-fetched raw artifact as-is (rather than a `Codec`-compressed or repackaged form) is only size-optimal because `exasol-local-runner`'s real archive happens to be minimal today. If a future `embed: true` resource's real archive carries extra content beyond what `resource_path` selects, embedding it as-is would carry that extra weight — the repackaging approach considered and deferred earlier in this design would need to be revisited then.

## Open Questions

- **Resolved:** on an unsafe version relationship (major-version mismatch, or the resolved runner older than the persisted marker), the launcher proceeds with the resolved runner anyway, logs a warning, and updates the persisted marker to match. This trades the old hard-guarantee ("never silently apply an unsafe change") for availability, since there's no fallback binary to keep using instead — closer to how a resource with no reconciliation at all (e.g. `tofu`) already behaves. `reconcileRunner`'s categorization logic (compatible upgrade / no-downgrade / major-gate / legacy-migration / forced-reconciliation-bypass) is retained for its *logging and marker-update* behavior, not as an enforcement gate.
- **Resolved:** the generator tool is `tools/resourceembedder`.
- **Resolved:** the persisted runner-version marker is a small separate file, `runner-version.json`, alongside `vm-state.json` in the runtime work directory — not a field on `vm-state.json` itself, since that file's schema is the external runner contract, not ours to extend.
