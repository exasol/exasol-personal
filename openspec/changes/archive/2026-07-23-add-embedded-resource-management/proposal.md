## Why

The Exasol Local VM runner is currently the only runtime dependency handled completely outside `internal/runtimeartifacts.Manager`: it's fetched at build time by a bespoke helper (`tools/localrunner`), baked into the binary via hand-written `//go:embed` files, and reconciled into the deployment directory by a parallel, ad hoc versioning system (`internal/localruntime.reconcileRunner`) that duplicates concerns the resource manager already solves for `tofu` (caching, checksum verification, staleness). This makes embedding a one-off, hard-to-extend mechanism rather than a first-class option available to any resource, and leaves no clean way to override what gets embedded for local development short of a CLI flag that bypasses the resource manager entirely.

This change makes "embedded in the binary" a first-class resource source inside the existing resource manager, driven declaratively from `resources.yaml`, with build-time embedding generated via `go generate` instead of a bespoke tool.

## What Changes

- Add a resource-level `embed: true` field to `ResourceDefinition` in `internal/runtimeartifacts/spec.go`. `exasol-local-runner` gets it; `tofu` does not (and should not, for binary size).
- `internal/runtimeartifacts.Manager` gains a package-level `Register`/lookup registry. Resolution becomes a strict either/or on the resource's `embed` flag: `embed: true` resources resolve exclusively from the registry (a miss is a hard error, never a fallback to network sources); everything else resolves exclusively from the existing Git/File/Http sources, unchanged. On an embedded hit, the looked-up bytes are written to the artifact slot a network fetch would otherwise populate, then the **existing, unmodified** extraction step runs — no new decompression/compression code, and it reuses the same `Extractor`s already used for network-fetched archives.
- Add a new, generic build-time generator tool (replacing `tools/localrunner`, which is deleted) that reads `resources.yaml`, fetches and checksum-verifies every `embed: true` resource via the existing `Manager`, and writes the raw fetched artifact bytes into build-tag-gated, fully generated (gitignored) `.go` files under `assets/resources/generated/`. Invoked via `go generate`.
- `cmd/exasol/main.go` gains one blank import (`assets/resources/generated`) to activate the embedded data for the shipped binary; the generator tool never imports it, which is what keeps build-time fetches honest (always real, always checksum-verified).
- The VM runner binary is no longer copied into the deployment directory at all: `internal/localruntime` resolves it via the resource manager and executes it directly from the shared cache on every invocation (its `cmd.Dir` is already the deployment's working directory regardless of where the binary itself lives, so no copy is needed). The existing semver-aware reconciliation *behavior* is preserved as domain-specific logic in `internal/localruntime` (upgrade on newer minor/patch, no downgrade, major-version gating, legacy-runner migration, the `EXASOL_LOCAL_FORCE_RUNNER_RECONCILIATION` escape hatch), but it now compares the currently-resolved runner's reported version against a small persisted version marker in the deployment's own state, instead of an installed binary's version — there is no more file to atomically replace.
- Update `doc/development.md` to describe the new generator + `go generate` workflow, removing references to `tools/localrunner`.

## Capabilities

### New Capabilities
- `embedded-resource-generation`: the `embed: true` schema field, the build-time generator tool, and the generated-and-gitignored `assets/resources/generated/` output that makes a resource's data available inside the compiled binary.

### Modified Capabilities
- `runtime-artifact-cache`: the manager gains an embedded-data resolution path, checked before the network sources, for resources marked `embed: true`.
- `exasol-local-deployment`: the "Launcher-owned local runtime" requirement's runner-staging scenario changes from "atomically write the embedded runner into the deployment directory" to "resolve the runner via the resource manager and execute it directly from the shared cache, no per-deployment copy." The version-compatibility scenarios (no downgrade, major-version gating, legacy-runner migration, forced-reconciliation bypass) are preserved in spirit but now compare against a persisted version marker in the deployment's state rather than an installed binary's version. On an unsafe version relationship (major-version mismatch, or the resolved runner older than the persisted marker), the launcher proceeds with the resolved runner anyway, logs a warning, and updates the marker — see `design.md` for the reasoning.

## Impact

- `internal/runtimeartifacts/spec.go`, `manager.go`: new schema field, new registry, modified `refresh()`.
- `assets/resources/generated/`: new, gitignored, generator-owned directory.
- `assets/localruntimebin/`: removed (generated blob + hand-written wrapper files no longer needed).
- `tools/localrunner/`: deleted, replaced by a new generator tool.
- `internal/localruntime/runtime.go`: removes the atomic candidate-file-replace machinery (`writeRunnerCandidate`, `replaceRunnerCandidate`, the per-deployment `RunnerPath` copy) since the runner is no longer copied anywhere; keeps the `blang/semver`-based categorization logic (compatible upgrade, no downgrade, major-version gate, legacy migration, forced-reconciliation bypass), re-based on a persisted version marker instead of an installed file's version. Its operations (`Prepare`, `Status`, `Stop`, `Destroy`, `HealthCheck`, `RunCommand`) become methods on a new exported `Runtime` struct (built via `New(deployment, manager)`) that bundles the deployment and the `*runtimeartifacts.Manager` together and resolves the runner (path + reported version) through it, keeping the `local-runtime-boundaries` delegation (VM-runner-specific mechanics stay inside `internal/localruntime`) intact while Manager construction stays at the backend-selection layer for every backend uniformly. An `EXASOL_LOCAL_RUNNER_OVERRIDE_PATH` environment variable lets `Runtime` bypass the manager entirely and use an arbitrary runner binary — needed since the runner is now `embed: true`-only with no non-macOS artifact, which would otherwise leave CI and non-Mac contributors unable to exercise this path at all.
- `internal/deploy/backend.go`, `local_backend.go`: `newDeploymentBackend` constructs a `*runtimeartifacts.Manager` for the local backend the same way it already does for tofu, and `newLocalBackend` wraps it together with the deployment into a `*localruntime.Runtime`, stored on `localBackend` and used by every method.
- `Taskfile.yml`, `.goreleaser.yaml`: replace `local-runner-placeholders`/`local-runner-stage` wiring with the new generator's `go generate` invocation, still triggered per-target in the release build, with the invoking tasks unsetting `GOOS`/`GOARCH` first so ambient cross-compilation targets don't apply to the generator's own `go run` invocation.
- `doc/development.md`: workflow section rewritten.
